Мій Проєкт: "BMI Health-cabinet"

Головна ідея проєкту — створити універсальний інструмент для розрахунку Індексу Маси Тіла (ІМТ), реалізувавши його бекенд одразу на трьох популярних Python-фреймворках: Flask, FastAPI та Django. Це дозволило мені порівняти їхні підходи, переваги та недоліки на практиці.

Аналіз Технологій

Проєкт був реалізований одразу на трьох Python-фреймворках для порівняння їхніх можливостей, переваг та недоліків у контексті мого завдання.

Flask

Плюси (де допомагав):

  • Мінімалізм: Ідеально для швидкого старту. Немає нічого зайвого, що дозволило швидко запустити базовий розрахунок ІМТ.
  • Гнучкість: Повний контроль над тим, які компоненти (наприклад, ORM) я хочу додати.

Мінуси (де заважав):

  • "Батарейки не включені": Для профілів користувачів та історії мені довелося б вручну налаштовувати ORM (як SQLAlchemy) та систему автентифікації.

FastAPI

Плюси (де допомагав):

  • Швидкість (async): Найкращий вибір для асинхронних запитів, наприклад, для миттєвої видачі порад.
  • Авто-документація: Вбудований Swagger/OpenAPI — це неймовірно. Економить години на тестуванні та описі API.
  • Валідація Pydantic: Чітка валідація вхідних даних (зріст, вага) прямо "з коробки".

Мінуси (де заважав):

  • Складність Async: Потрібно було слідкувати за асинхронним кодом, що трохи складніше для простого CRUD.
  • Екосистема: Молодший за інших, тому менше готових плагінів, ніж у Django.

Django

Плюси (де допомагав):

  • "Все включено": Вбудована адмін-панель, ORM та система автентифікації — ідеально для профілів, історії вимірювань та плейлістів порад.
  • Безпека: Багато вбудованих засобів захисту (CSRF, XSS), про які не потрібно думати.

Мінуси (де заважав):

  • Монолітність: Надто "важкий" для простого розрахунку ІМТ. Його потужність була надлишковою для мікросервісів.
  • Менша гнучкість: "The Django way" — іноді доводиться робити речі так, як хоче фреймворк.
